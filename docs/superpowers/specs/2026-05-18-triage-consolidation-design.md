# Triage Consolidation Design

**Date:** 2026-05-18
**Status:** Approved

## Context

The SPA currently has two separate pages for the sample collection workflow:

- `/triage` — shows Arrived patients (send to waiting room) and Waiting patients (print barcodes)
- `/waiting-room` — shows Waiting patients (call patient) and provides the collect sample form

This duplication forces lab technicians to switch between pages for tasks that are part of the same workflow. The goal is to merge both into a single `/triage` page that covers the full lifecycle: Arrived → Waiting → Called → Collected.

Additionally, the current "Print Labels" button calls `window.print()` on the full page, which prints the entire UI instead of just the barcode labels.

## What Changes

### 1. Unified `/triage` page

A single page replaces both `/triage` and `/waiting-room`. It shows all patients across all active statuses in a grouped, filterable list.

**Layout: Filtered List with section groups**

- A filter bar at the top with pill tabs: **All** · **Arrived** · **Waiting** · **Called**
- Each tab shows a live count badge
- Selecting a tab hides the other status groups; "All" shows all three with section headers
- Default tab on load: **All**

**Patient card rows**

Each row shows:
- Patient ID (monospace)
- Status timestamp and sample count in secondary text
- Status badge (color-coded: green/amber/blue)
- A **⋮ kebab menu button** at the far right

The left border of each card is color-coded by status (green = Arrived, amber = Waiting, blue = Called).

**Kebab menu — contextual actions per status**

| Status   | Menu items                              |
|----------|-----------------------------------------|
| Arrived  | Send to Waiting Room                    |
| Waiting  | Print Label · (divider) · Call Patient  |
| Called   | Record Collection                       |

"Record Collection" expands an inline form below the card (same behaviour as the current waiting-room page).

### 2. Isolated barcode print modal

Clicking **Print Label** from the Waiting kebab menu opens a modal overlay instead of calling `window.print()` on the full page.

The modal contains:
- Title: "Print Labels — {patientId}"
- Subtitle: "{n} samples · barcodes generated by the system"
- One row per sample: tube type label · CODE128 barcode SVG (rendered by JsBarcode) · barcode value in monospace
- **Cancel** and **Print** buttons

Clicking **Print** calls `window.print()`. `@media print` CSS hides everything except the modal content, so only the label rows are sent to the printer.

### 3. Route cleanup

- `/waiting-room` route is removed from the router
- A redirect from `/waiting-room` → `/triage` is added so any saved links still work
- The sidebar navigation link for "Waiting Room" is removed; the existing "Triage" link remains

## Component Architecture

### New / replaced components

| Component | Change | Notes |
|-----------|--------|-------|
| `triage.component.ts` | Rewrite | Replaces old two-section layout; owns filter state and the full patient list |
| `patient-row.component.ts` | New | Single card row used for all statuses; receives `CollectionRequestSummary` + emits action events |
| `kebab-menu.component.ts` | New | Generic ⋮ dropdown; receives an `actions: {label, icon, event}[]` input |
| `print-labels-modal.component.ts` | New | Wraps existing `PrintLabelsCardComponent` in a modal with print isolation |
| `collect-sample-form.component.ts` | Move | Already exists in waiting-room; moved/re-used in triage |
| `arrived-patient-card.component.ts` | Delete | Replaced by `patient-row` |
| `preparing-patient-card.component.ts` | Delete | Replaced by `patient-row` |
| `waiting-room.component.ts` | Delete | Page removed |

### Data flow

`TriageComponent` loads all three status groups on init (or on filter change) by calling the port:

- `loadArrived()` → Arrived group
- `loadQueue()` → Waiting group  
- `loadCalled()` → Called group (new port method needed — `GET /collection-requests?status=Called`)

Actions dispatch through the existing port methods:
- `moveToWaiting(id)` — Arrived → Waiting
- `getSamples(id)` — fetches samples for the print modal
- `callPatient(id)` — Waiting → Called
- `recordCollection(id, params)` — Called → Collected

After any action, only the status groups that could have changed are refreshed (e.g., "Send to Waiting" re-fetches Arrived + Waiting; "Call Patient" re-fetches Waiting + Called; "Record Collection" re-fetches Called only).

### Backend change

`GET /collection-requests` already accepts a `status` query param. The `loadCalled()` port method is a new frontend-only addition calling the existing endpoint with `status=Called`. No backend changes needed.

## Print Isolation

```
@media print {
  body > * { display: none; }
  .print-labels-modal { display: block !important; }
}
```

The modal component adds/removes a `print-labels-modal` class on the `<body>` while open. This hides everything except the modal when the browser print dialog fires.

## Navigation / Routing

```
/triage          → TriageComponent (unified page)
/waiting-room    → redirectTo: /triage (permanent redirect)
```

Sidebar: remove "Waiting Room" entry; keep "Triage".

## E2E Tests

Existing triage and waiting-room Playwright specs are merged into a single `triage.spec.ts` covering:

1. Full workflow: Arrived → Send to Waiting → Print Label (modal appears) → Call → Record Collection
2. Filter tabs narrow the list to the correct status group
3. Kebab menu shows only the valid action(s) for each status
4. Print modal: opens, shows correct samples, closes on Cancel
5. Role guard: non-LabTechnician/ITAdmin roles are redirected to `/unauthorized`
6. `/waiting-room` redirect: navigating to `/waiting-room` lands on `/triage`
